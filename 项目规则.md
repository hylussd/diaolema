# 钓了吗 — 项目规则

> 每次项目更新时，所有相关方必须遵守的规范。

---

## 一、文档同步规则

### 每次提交必须包含以下检查：

**1. 功能变更 → 必须更新**
- 新增功能 → 同步到 `README.md` 更新日志 + `01-方案/分阶段规划.md`
- 新增接口 → 同步到 `05-开发/API接口设计.md`
- 新增表/字段 → 同步到 `05-开发/数据模型.md`
- 新增文件 → 同步到 `05-开发/目录结构.md`
- 架构变更 → 同步到 `05-开发/技术设计文档.md`

**2. 提交格式要求**
```
feat: <功能名称>
fix: <修复内容>
docs: <文档同步>
chore: <其他>
```

**3. README.md 更新日志格式**
```markdown
### YYYY-MM-DD — <版本/阶段>
- <功能1>
- <功能2>
- 新增：backend X文件 / frontend Y文件 / Z个新页面
```

---

## 二、代码规范

### 后端（Python FastAPI）
- 所有接口返回格式统一：`{"code": 0, "data": ..., "msg": ""}`
- 新增模型必须在 `init_db.py` 中注册建表
- 新增路由必须在 `main.py` 中注册
- 敏感信息（API Key等）写入 `.env`，`.env` 不提交到 Git
- 不引入不必要的外部依赖，规则引擎纯本地计算

### 前端（微信小程序）
- WXML 模板中禁止使用 ES6 箭头函数（`=>`）
- 复杂表达式移到 JS 的 `data` 或工具函数中
- TabBar 图标资源：81×81px PNG，存 `assets/icons/`
- 所有 API 请求封装到 `services/` 目录，禁止直接 `wx.request`

---

## 三、Git 规范

### 分支策略
- 单一主分支 `main`，所有变更直接合并到 `main`

### Commit 规范
- 功能完成必须先 commit 再 push
- 不允许未测试的代码推送到 `main`
- `.env`、`.DS_Store`、`__pycache__/`、`.pyc` 不提交

### .gitignore 必须包含
```
.env
.DS_Store
__pycache__/
*.py[cod]
*.db
data/
```

---

## 四、项目阶段流转规则

### 阶段进入条件
- P0 完成 → 策划虾评审 → Ticket 标记 DONE → 才能启动 P1
- P1 完成 → 文档全部同步 → Ticket 标记 DONE → 才能启动 P2
- 依此类推

### Ticket 必须包含
- 状态（RUNNING / DONE / BLOCKED）
- 功能清单（逐项标注 ✅/⬜）
- 产出物路径
- 完成时间

---

## 五、产出物检查清单

每次阶段完成，交付前必须确认：

- [ ] 所有代码文件已实现
- [ ] 所有文档已更新（见第一节）
- [ ] README.md 更新日志已写
- [ ] `.tickets/` 下 ticket 已标记 DONE
- [ ] Git 已 commit + push
- [ ] 新增功能未引入编译错误
- [ ] `.env` 未泄露到 Git
